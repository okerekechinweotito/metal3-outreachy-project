Automate the deployment of bare metal infrastructure using Kubernetes.

Metal³ is an open-source project collaboration between OpenStack Foundation and Kubernetes Community. Metal³ is itself a Kubernetes application, so it runs on Kubernetes, and uses Kubernetes resources and APIs as its interface.

Metal³.io Overview

Metal³ makes it easier for organizations to deploy and manage their applications at scale. 

        
        Metal³ Prerequisites

Why Metal³.io

There are a number of great open source tools for bare metal host provisioning, including Ironic. Metal3.io aims to build on these technologies to provide a Kubernetes native API for managing bare metal hosts via a provisioning stack that is also running on Kubernetes. We believe that Kubernetes Native Infrastructure, or managing your infrastructure just like your applications, is a powerful next step in the evolution of infrastructure management.

The Metal3.io project is also building integration with the Kubernetes cluster-api project, allowing Metal3.io to be used as an infrastructure backend for Machine objects from the Cluster API.

One of the key benefits of using Metal³ is the ability to manage infrastructure just like applications. This means that users can use familiar tools and interfaces to manage their bare metal hosts, making it easier to integrate these hosts into their existing workflows. Additionally, by running the provisioning stack on Kubernetes, users can take advantage of the scalability and flexibility offered by this platform.

The goal of Metal³ is to make it easier to manage and deploy bare metal servers at scale, similar to how cloud providers manage virtual machines. It provides a solution that is scalable and suitable for use in various use cases, including edge computing, high-performance computing, and traditional data centers.

Components

Baremetal Operator (BMO)

a controller that manages the BareMetalHost resources and interacts with the hardware using protocols like Redfish. This is like a switchboard that lets you turn on and off your computers and make them do things. For example, you can use it to make your storage computer save data or delete data, or make your application computer run a program, or stop a program... Read more

Cluster API Provider Metal³ (CAPM3)

a component that integrates Metal³ with Cluster API, a framework for declarative Kubernetes cluster management. This is like a guidebook that tells you how to use your computers to do different things. For example, it can tell you how many storage computers you need for saving data, how many application computers you need for running programs, etc... Read more

BareMetalHost

a custom resource that represents a physical server and its inventory data. This is like a name tag that you put on each computer to tell what kind of computer it is and what it can do. For example, you can name a computer as “storage”, “application”, “internet”, etc... Read more

Hardware Classification Controller (HCC)

a controller that classifies BareMetalHosts into user-defined groups based on hardware details. This is like a sorter that helps you organize your computers into different groups based on their name tags. For example, it can sort all the storage computers together or all the internet computers together... Read more

IP Address Manager (IPAM)

a component that assigns IP addresses to BareMetalHosts from a pool of available addresses. This is like a phone book that gives each computer a unique number so you can call them and talk to them. For example, you can use it to tell one internet computer to send data to another internet computer or receive data from another internet computer... Read more

Metal³ Hardware Inventory Record (M3HIR)

a custom resource that stores hardware inventory data collected from BareMetalHosts. This is like a diary that keeps track of all the details about your computers and what they are doing. For example, it can record how much data each storage computer has saved or deleted or how fast each application computer has run a program... Read more

    Goals

  • To provide a framework that is scalable, vendor-neutral, and accessible to users of any skill level.

  • Metal³ aims to reduce the complexity and cost of managing bare metal infrastructure by leveraging Kubernetes to automate many of the common tasks associated with managing servers.

  • Metal3 aims to provide a consistent and declarative API for managing bare metal infrastructure.

    Missions

  • To make the deployment, configuration, and management of bare metal infrastructure as easy as cloud infrastructure.

  • To provide a solution that is suitable for use in various use cases, including edge computing, high-performance computing, and traditional data centers

  • To automate the lifecycle management of bare metal hosts, including provisioning, configuration, and decommissioning

Quick Code Overview

  • setup
  • tear down
  • env var
              
                clone 
                  metal³-dev-env and run 

                $  make 


  
              The Makefile runs a series of scripts, described here:


  
              01_prepare_host.sh - Installs all needed packages.


  
              02_configure_host.sh - Creates a set of VMs that will be managed as if they were bare metal hosts. It also downloads
              some images needed for Ironic.


  
              03_launch_mgmt_cluster.sh - Launches a management cluster using minikube or kind and runs the baremetal-operator on that
              cluster.


  
              04_verify.sh - Runs a set of tests that verify that the deployment completed successfully.


  
              When the environment setup is completed, you should be able to see BareMetalHost (bmh) objects in Ready state.
              
              
                To tear down the environment, run 

                $ make clean 


  
              Note

  
              When redeploying metal³-dev-env with a different release version of CAPM3, you must set the FORCE_REPO_UPDATE variable
              in config_${user}.sh to true.


  
              Warning

  
              If you see this error during the installation:

  
  
              error: failed to connect to the hypervisor \ 

              error: Failed to connect socket to '/var/run/libvirt/libvirt-sock': Permission denied


              You may need to log out then login again, and run make clean and make again.
  
  
              
              
                To set environment variables persistently, export them from the configuration file used by metal³-dev-env scripts: 


                $ cp config_example.sh
                config_$(whoami).sh
                

                $ vim config_$(whoami).sh
